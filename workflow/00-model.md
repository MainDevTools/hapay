# Модель двох сесій: Диригент + Виконавець

**Статус: чернетка.** Дві окремі сесії Claude Code з різними ролями; ви — між ними.

## Навіщо саме дві сесії

Розділення «думати» і «робити» дає те, чого одна сесія не дає:
1. **Чистий контекст у Виконавця** — жодного накопиченого «сміття» довгих розмов, менше дрейфу від інваріантів.
2. **Незалежне рев'ю** — Диригент не писав код, тож не виправдовує його (захист від «модель хвалить свій же код»).
3. **Різні режими** — планування рефлексивне й адверсарне; виконання — heads-down.

## Ролі

| Диригент (думає) | Виконавець (робить) |
|---|---|
| дорожня карта, пріоритети, «що далі» | сфокусована реалізація одного брифа |
| формулювання задачі + **DoD** (критерій приймання) | тести, VCR-касети, канарки |
| заземлення у спеці (`../0X.md`, §-посилання) | `probe.py`, реальні крамниці — **ground truth** |
| ловить scope-creep, тримає інваріанти | звітує, що в реальності **насправді** є |
| **незалежне рев'ю** діффа Виконавця | evidence: дифф, вивід тестів, `/verify` |
| оновлює спеку/ADR при дрейфі реальності | оновлює касету/селектор при редизайні крамниці |

**Ви — шина повідомлень + фінальний арбітр.** Тримаєте *намір* («навіщо») і приймаєте остаточно. Жодна сесія цього не замінює.

## Вирішальний принцип: сесії спілкуються через РЕПО, не через пам'ять

Дві сесії **не бачать пам'яті одна одної**. Канал між ними — (а) ви, (б) файли в репо. Проектуємо на (б), щоб ви носили менше:

- Диригент пише бриф → `tasks/SNN-*.md` (не усно).
- Виконавець читає бриф, робить, **дописує «Outcome»** у той самий файл (що зроблено, тести, що виявив у реальності).
- Обидві сесії читають `CLAUDE.md`, `docs/` (спеку), **фактичний код + вивід тестів** — не переказ.
- Рішення → журнал `../00-index.md` / `docs/adr/`.

## Петля (один цикл роботи)

1. **Диригент** із дорожньої карти (`../11-mvp.md` §11.2) обирає наступне; пише бриф: ціль + §-посилання + **DoD** + out-of-scope + guardrails.
2. **Ви** передаєте бриф Виконавцю («прочитай `tasks/SNN.md`»).
3. **Виконавець**: план-мод → код → тести зелені → дописує «Outcome» + **evidence**.
4. **Диригент** рев'юїть Outcome проти DoD (свіжими очима) → «прийнято» або «розбіжність».
5. **Реальність-фідбек:** якщо Виконавець виявив дрейф крамниці / інший селектор — це йде **назад у Диригента**, який оновлює спеку `../03-dzherela.md` §3.3 і план. **Контракт не відстає від реальності.**
6. Наступний бриф.

## Пастки й обхід

| Пастка | Обхід |
|---|---|
| Диригент планує у вакуумі на застарілих припущеннях | реальність-фідбек (крок 5) **обов'язковий**; спека жива |
| Хендофф усний/розмитий → Виконавець вгадує | бриф — **самодостатній файл** із DoD |
| Виконавець накопичує довгий контекст → дрейф від інваріантів | **свіжа сесія Виконавця** на задачу (або кілька) — головна вигода моделі |
| Ніхто не перевіряє | Виконавець дає evidence → Диригент рев'юїть проти DoD → ви арбітр |
| «Один замір оманливий» (діє на **обидві** сесії) | правило «перевір із першоджерела, не роби висновок з одного скану» — в `02-invariants.md` для обох |

## Відкриті питання (доробити)

- Як часто освіжати сесію Виконавця — на кожну задачу чи на 2–3?
- Чи потрібна третя роль «рецензент», чи Диригент її покриває?
- Де межа: коли Диригент оновлює спеку сам, а коли делегує Виконавцю.
