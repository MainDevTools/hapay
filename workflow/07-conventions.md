# Git-безпека: щоб не було витоку інформації

## Принцип 0 — історія незворотна й може стати публічною

- Що **раз закомічено й запушено — лишається в історії назавжди**, навіть після видалення файлу наступним комітом. Видалення ≠ зникнення.
- **O8 (open-source методології) ще відкрито** → пиши так, **ніби історія вже публічна**. Секрет у історії = **скомпрометований** секрет, не «потім приберемо».
- Прямий зв'язок з інваріантом **B (лише факти, не байти чужих фото)**: заборона стосується й git — чужий контент не має потрапляти в історію.

## 1. Що НІКОЛИ не комітити (перевір `git diff --staged` перед кожним комітом)

- **Секрети/креденшели:** `.env`, токени (Telegram bot token, Turso/libSQL, Cloudflare D1/Workers, афіліат-ключі), паролі, приватні ключі (`*.pem`, `*.key`, `*.p12`), сервіс-акаунти (`*credential*.json`).
- **Дані збору:** файл БД (`*.db`, `*.sqlite*`, `-wal`/`-shm`), дампи, зібрані ціни.
- **Юр-чутливе (інв. B):** байти чужих фото, кешований HTML/сирі payload'и крамниць, будь-який завантажений чужий медіа-контент.
- **PII:** реальні Telegram `user_id`, watchlists, особисті дані в фікстурах/логах.
- **Локальне:** `.venv/`, `__pycache__/`, IDE-конфіги, `*.log`, кеш тулів.

## 2. `.gitignore` — перша лінія оборони

Створити **до першого `git add`** (він уже є в корені: [`../.gitignore`](../.gitignore)). Покриває все з п.1 за замовчуванням. Але: **`.gitignore` не захищає вже-відстежувані файли** — якщо файл раз потрапив під контроль, ігнор його не прибере. Тому оборона №1 — не додати, а не «вичистити потім».

## 3. Секрети — лише через середовище

- Ключі — у **змінних середовища / `.env`** (git-ignored). **Ніколи не хардкодити** ключ у код.
- У репо — тільки **`.env.example`** з **назвами змінних без значень** (документація, не секрет). Створити, коли з'являться перші секрети.
- Секрети деплою (GitHub Actions/Workers) — у **secrets-сховищі платформи**, не в репо (central-light §8.10.1).

## 4. Pre-commit секрет-скан (гейт №27)

Хук, що **блокує коміт** при патернах секретів (gitleaks / trufflehog або мінімальний regex: `token=`, `api_key`, `password`, приватні ключі, довгі base64/hex). Специфікуємо зараз, вмикаємо з тулінгом. Порушення = блок (узгоджено з #28 CI-гейт / #29 порушення=блок мержу).

## 5. Якщо секрет усе ж потрапив у коміт — порядок дій

1. **Спершу РОТУЙ секрет** (відклич токен/змінь ключ). Вважай його скомпрометованим — чистка історії **не гарантує**, що ніхто не встиг клонувати/закешувати.
2. Аж потім вичисти історію (`git filter-repo` або BFG) + `--force` push, якщо вже запушено.
3. Признач інцидент оператору (як BLOCKED). Занотуй у [10-postmortems] / captain-log.

> Порядок саме такий: **ротація перша**, бо переписування історії ненадійне проти вже-клонованого. CC **не** робить force-push/переписування історії без явного «так» оператора.

## 6. Remote — приватний за замовчуванням

- Поки **O8 не вирішено** — репозиторій **приватний**. Публічний remote/публікація історії = **рішення власника (🧭)**, не CC.
- Перед будь-яким `git remote add` чи push у публічне — **стоп, підтвердження оператора** (правило «публікація = дозвіл оператора»).

## 7. Гігієна комітів

- **Не класти** секрети / особисте / внутрішні URL у **повідомлення комітів** і **назви гілок** — вони теж в історії та в remote.
- **Оглядай `git diff --staged` свідомо** перед комітом. Уникай сліпого `git add -A` / `git commit -a` на ранньому етапі — додавай **явно перелічені** файли, щоб не затягнути зайве.
- Атомарні коміти (один сенс — один коміт); коміт кожного зеленого кроку (чекпойнт-дисципліна, `04-captain-log`).

## 8. VCR-касети / фікстури — окрема пильність

Касети комітяться **для детермінованих тестів (T5)**, але **лише очищені**:
- **замаскувати** заголовки авторизації, куки, токени в URL/тілах;
- **обрізати** тіла до мінімуму, потрібного тесту;
- **виключити байти чужих фото / великий копірайт-контент** (інв. B).

Неочищений сирий захват — **не в git** (`tests/cassettes/raw/` ігнорується). Pre-commit скан (п.4) перевіряє й касети.

## 9. Чек-ліст перед першим пушем

- [ ] `.gitignore` на місці **до** першого `git add`.
- [ ] `git ls-files` — переглянути, що взагалі відстежується (нема `.env`/БД/секретів/чужих байтів).
- [ ] `git log` / `git diff --staged` — жодного секрету в діффах і повідомленнях.
- [ ] Remote приватний (або remote ще нема).
- [ ] Секрети — лише в `.env` (ignored) і secrets-сховищі платформи.

## Решта конвенцій — заповнити з кодом

*Стаби (не churn — місце під код-таймові правила, щоб не загубити):*
- **repo-map** — карта каталогів/модулів, коли з'явиться структура коду.
- **Definition of Ready** — коли бриф `tasks/SNN` готовий до Виконавця (ціль+DoD+§-контекст+guardrails однозначні).
- **Протокол «застряг»** — `BLOCKED: <що/чому>` в §Outcome + стоп (вже в `02`/CLAUDE.md; тут розширити прикладами).
- **Стандарт рев'ю** — Диригент звіряє Outcome з DoD; **обов'язкове людське рев'ю юр-інваріантів §7.4 і статутної формули §5** (#25).
- **Як оновлюється спека** — розходження з реальністю → відповідний § + рядок у журнал `00-index` (реальність-фідбек P3).
