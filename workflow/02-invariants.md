# Тверді інваріанти й правила (для обох сесій)

**Статус: чернетка → стане `CLAUDE.md` у корені репо, коли з'явиться код.**
Це те, що **ніколи** не порушується, незалежно від задачі. Обидві сесії (Диригент і Виконавець) тримають це в голові.

## A. Дані та гроші

1. **Гроші — лише цілі копійки** `*_kop BIGINT` (§6.1); Postgres відхиляє не-ціле на рівні типу. Жодного `REAL`/`NUMERIC`/float для грошей.
2. **`price_snapshot` — append-only** (ABORT-тригери на UPDATE/DELETE). Історія незмінна (§6.3).
3. **Крос-крамничного зіставлення НЕМАЄ** — товар унікальний у межах джерела (§6.1, §12). Не «матчити той самий товар між крамницями».
4. **Час:** зберігання ISO-8601 UTC; цивільні дати (30-денне вікно) — `Europe/Kyiv` через IANA-таймзону, не фіксований офсет (DST, §5.2).

## B. Юридична чистота (не порушувати — це суть продукту)

5. **Зберігаємо лише ФАКТИ:** ціна, назва, URL, час, **характеристики** — пари назва-значення технічних фактів зі спец-таблиць («Діагональ екрана» → «43"»); завжди з провенансом (крамниця+URL+дата). *(Розширення затверджено оператором 2026-07-24, S12.)* **НЕ** копіювати описи (зв'язний маркетинговий текст — ані байта, ніде, включно з note) й **байти** фото (§7.4).
6. **Фото — тільки вказівник:** `image_url` (hotlink/фід) + `image_blurhash`. **Ніколи** не завантажувати/зберігати/перевіддавати байти твору й не ставити сторонній image-proxy (§7.4).
7. **Не обходити техзахист:** CAPTCHA/авторизацію/активні блокування не долати. Прийняти авто-куку — можна (§7.4, §3.5).
8. **Формулювання бейджа — фактологічні**, не оціночні (ст.277 ЦК, §5.4).

## C. Детекція

9. **Статутний `reference_kop` читає СИРИЙ `price_snapshot`**, ніколи похідний `price_daily` (§5.2, §8.6) — це доказ для ст.277.
10. Числа детекції — з датованої `detection_config`, **не з коду** (§11.6). Зміна порога = рядок конфігу.

## D. Архітектура / код

11. **БД — чистий PostgreSQL 16** (§6; ship на Neon-free, T11): `price_snapshot` — звичайна таблиця + покривний `ix_ps_prod_window`; FTS — `tsvector`+GIN; append-only — тригер + `REVOKE UPDATE/DELETE`. Доступ — через пул (`statement_timeout`/`lock_timeout`); FK чинні завжди. *(Timescale hypertable/cagg/компресія — `0002` scale-upgrade, не зараз.)*
12. **Краул — поза транзакцією:** буфер у пам'яті → коротка write-транзакція (§8.6). Не тримати транзакцію через мережевий I/O.
13. **Топологія — central-only** (GitHub Actions-збір → central Postgres/Timescale → Telegram Mini App); **desktop-local відкинуто (O1 закрито)** (§8.10). Шари 1–2 спільні, один краул на всіх.

## E. Робоча дисципліна (проти режимів відмови моделі)

14. **Не робити висновок з ОДНОГО скану/проби/джерела.** Currency-факти — з першоджерела + повтор (≥2 узгоджені). Урок Pethouse/Lume: один замір двічі вводив в оману (§10.9, журнал §00).
15. **Evidence, не «я перевірив».** Будь-яке «готово»/«працює» — з діффом, тестами, `/verify`.
16. **Реальність → спека → код.** Якщо крамниця/факт розходиться зі спекою — спершу оновити спеку (§3.3 тощо), потім код.
17. **Вузький скоуп.** Одна задача = один бриф. Не роздувати; фільтр — «чи дійсно підсилює».
18. **Інваріанти — як тести, не лише текст.** A1, A2, B5/B6 мають бути виконуваними guardrail-тестами, що червонять CI при порушенні.

## E-bis. Режими відмови Claude Code (знати обом сторонам)

Доведені **в самій цій розробці** — не теорія:
- **Над-продукція / churn.** CC схильний генерувати більше, ніж треба (спека виросла вдвічі, потім CC сам чистив свій же churn). → **Дефолт = мінімум, що вирішує; більше — на запит.** Роздування = дефект.
- **Не зупиняється сам.** На «ще» CC виконує, **не відмовляючи**, навіть за спадної цінності. → **«Enough»-протокол** (`08-interaction-protocol.md` §1): CC **сам сигналить «вичерпано»** і не продовжує без явного «так, ще».
- **Над-впевненість з одного джерела.** CC двічі зробив висновок з одного заміру й помилився (Pethouse, Lume). → маркери впевненості (`08` §2) + правило ≥2 (E14).
- **Дрейф від інваріантів на великому контексті.** → інваріанти як тести (E18), свіжа сесія Виконавця на задачу.
