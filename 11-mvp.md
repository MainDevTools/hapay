# Розділ 11. MVP і дорожня карта

## 11.1. Принцип M1 — вузько й до кінця

Одна вертикаль, кероване число крамниць, повний вертикальний зріз (збір → історія → детекція → UI), доведений до працюючого бейджа. Ширина додається лише після того, як ядро працює.

## 11.2. Скоуп M1

| Параметр | M1 |
|---|---|
| Вертикаль | **Зоотовари** (§2.3) |
| Категорії | **сухий корм · коти + собаки** (1–2 підгрупи) |
| Крамниці | **6:** MasterZoo, Zootovary.ua, Petslike, Pethouse, PetChoice, Zoobonus (§3.3) |
| Обсяг | ~200–400 знижок у показі (discovery); baseline-стеження всієї категорії ~500–1500 товарів (§10.8) |
| Стадія A | заявлені знижки — **день 1** |
| Стадія B | верифікований бейдж — через ~4 тижні живого збору (ретро-набудова для зоо-ніші майже нічого не дає — §4.6/§10.7; на неї не закладаємось) |
| UI | онбординг-майстер (§9.6) → категорія → перелік → фільтри → картка з графіком історії |
| Каденс | discovery 2×/добу · baseline 1×/добу (§10.8) |

**Оцінка часу:** 4–6 тижнів на реалізацію ядра — повторні проби 2026-07-08/09 (§3.3) показали, що три адаптери складніші за «SSR/CSS»: Zootovary — CAPTCHA/реверс XHR, Petslike — гідратація/JSON-LD, Zoobonus — уточнення поверхні; плюс Horoshop-клас (MasterZoo/Petmarket), каскад §4, схема §6, тести §8.8, desktop-UI. **Плюс ~4 тижні живого збору** до перших верифікованих бейджів (архівного шорткату немає).

**Порядок реалізації (за спаданням ROI):** (0) **`probe.py` (§3.9/§8.8) — перший рядок коду**: валідує ризик №1 (збір) на живих даних до адаптер-класів; (1) `0001`-схема §6 + connection-factory §8.2; (2) **два доведені SSR-адаптери — Pethouse, PetChoice** (працюють plain-GET) на **VCR-касетах** (§8.8) — щоб замкнути цикл збір→історія→бейдж на тому, що реально віддає ціни; (3) **Horoshop-клас-адаптер** — найвища ROI-точка проекту: одна робота масштабується на 8000+ UA-магазинів на цій платформі (§3.6); (4) решта крамниць — за probe-гейтом. Не починати з крамниць, що воюють з anti-bot.

> **Топологія M1 — рекомендовано central-light (§8.10):** збір запускати на **GitHub Actions** (безкоштовно, безперервна щільна історія → бейдж спрацьовує), фронт — **Telegram Mini App + бот** (category-first UI, нуль інсталяції). Це закриває Тези 3/4 слабкостей (§13/§14) дешевше за desktop; Шари 1–2 (адаптери §4, схема §6, детекція §5) — ті самі, міняється лише *де* збір і *чим* споживання. Desktop-.exe (решта §8) — валідна max-privacy-альтернатива, не первинна.

> **Go/no-go гейт після probe-фази (2 тижні).** Перш ніж інвестувати в повний набір адаптерів — 2 тижні probe-скрипта (§3.9) + 2 SSR-адаптери на живих даних. Гейт: чи з'являються зелені/жовті бейджі взагалі, чи тримається збір, чи є ≥100 знижок/добу. Не пройшов — переоцінити скоуп (менше крамниць / інша вертикаль / пивот §12.4) **до**, а не після, місяців розробки.

## 11.3. Критерії успіху M1

| Критерій | Ціль | Що означає провал |
|---|---|---|
| Покриття збору | ≥ 5/6 крамниць стабільно віддають ціни | адаптери занадто крихкі |
| Обсяг знижок | ≥ 100 активних знижкових позицій щодня в категорії | категорія «суха» на акції |
| Верифікація | ≥ 30% заявлених знижок отримують зелений/жовтий бейдж | історії/сигналу замало |
| Викриття фейків | помітна частка «−N% заявлено / ~0% реально» (жовтий) | або фейків мало, або детекція не ловить |
| Незалежна валідація фейк-частки | ручна вибірка ~100 знижок звірена з нашим бейджем; частка фейків узгоджується з нашим виміром ±розумно | теза «60–70% фейків» (§1) висіла б на одному зовнішньому джерелі (Lume) — M1 має дати **власний** незалежний замір |
| Свіжість | оновлення 2×/добу без банів > тижня | anti-ban/каденс не тримає |
| **Точність бейджів** | ручна звірка ≥ 20 випадкових verified/pumped проти сирої історії: ≥ 90% коректні | soft-break пише правдоподібне сміття (§10.9) або детекція бреше |

**Якщо M1 показує, що зелених бейджів майже нема** (усі знижки або фейкові, або мілкі) — це валідний результат, що переорієнтовує продукт (розділ 12): цінність зміщується з «знаходь знижки» на «викривай фейки» (жовтий бейдж як головна фіча).

## 11.5. Бюджет (орієнтовно, central M1)

> ⚠ **O1 закрито: топологія central-only** (§8.10) — «локальний, без хостингу» більше не діє. Збір — GitHub Actions (безкоштовно на публічному репо), але **БД — керована PostgreSQL/TimescaleDB** (не ~0).

Основна вартість — **час розробки**; інфра невелика, але не нульова:
- розробка ядра M1 — 2–4 тижні;
- інфра — **керована Postgres/Timescale** (free-tier / копійчаний VPS для self-host) + GH Actions cron (безкоштовно) + Telegram (безкоштовно); опційно дешевий проксі, якщо крамниця почне блокувати. Порядок — десятки $/міс, не сотні;
- підтримка — закладати ~кілька годин/квартал на фікс адаптерів (редизайни, §10.6) + базовий DB-ops (бекап/PITR — §8.6-central).

Central-топологія свідомо **бере** витрати хостингу й ширшу юр-експозицію (§7.5 публічний показ) в обмін на щільну історію (зелений бейдж §5.3), накопичуваний рів (§12) і нуль-інсталяційну дистрибуцію (Mini App). Це прямий наслідок O1.

## 11.6. Контракт «без літералів у коді»

Усі числа детекції, пороги, каденс, мапінги категорій — у датованих config-таблицях (`detection_config`, `source_category_map`, `app_config`), не в Python-коді. Зміна порога знижки чи вікна = новий рядок конфігу, не реліз.

> **Уточнення: recompute vs «історичні події зберігають розрахунок» (усунення двозначності з §8.4).** `detect_pass` перераховує щоскан **лише активні** події (`ended_at IS NULL`) — і рахує їх **поточним** чинним рядком `detection_config` (зміна порога діє на живі знижки з дня зміни). **Закрита** подія (`ended_at` виставлено) — **незмінна**: її `*_pct`/`badge_state`/`reference_kop` заморожені на момент закриття й `detect_pass` їх не чіпає. Провенанс: `discount_event.computed_at` разом із датованістю `detection_config` (`valid_from`/`valid_to`) однозначно визначає, який конфіг застосовано (рядок, чинний на `computed_at`). Тому «історичні зберігають розрахунок» = **закриті**; «конфіг датований» дає відтворюваність без окремого FK на config.
