# Розгортання на Hetzner

> ⚠ **Написано наосліп.** У пісочниці розробки немає ні Docker, ні WSL, ні Postgres —
> локально це не перевірялось. **Перший запуск на сервері й буде тестом.** На чистій
> машині це прийнятно; на машині з даними — ні.

## Що тут крутиться і чого тут НЕМАЄ

| | Де |
|---|---|
| **API** (`api.main:app`) | тут, у Docker |
| **Caddy** (TLS, зворотний проксі) | тут |
| **Postgres** | **на Neon, не тут** — свідомо |
| **Колектор** | поки на GitHub Actions (безкоштовно) |

**Чому база лишається на Neon.** 2026-07-17 ми знищили історію цін (T13), і єдине, що
дає шанс її повернути, — point-in-time restore керованої бази. Своя база на цьому ж
сервері означає, що страхівка залежить від того, чи не помилився я в `backup.sh`.
Neon безкоштовний і дає PITR — переносити базу сюди варто лише коли обсяг переросте
кероване (≈180 ГБ/рік на масштабі 5000 крамниць), і тоді бекапи стають критичним шляхом.

## Порядок

**Твої кроки** (CC не має права: акаунти, оплата, креденшели):

1. Hetzner: акаунт + верифікація → Cloud Console → **New project** → **Add server**
   - Location: Falkenstein / Nuremberg · Image: **Ubuntu 24.04** · Type: **CX22**
   - **SSH key** — свій; пароль root **не** вмикати
   - **Backups: увімкнути** (+20% до ціни — знімок машини)
2. Домен → DNS **A-запис на IP сервера**. Зроби це **до** підняття Caddy: у Let's Encrypt
   є ліміт невдалих спроб, і зайві провали доведеться пересиджувати.
3. На сервері:
   ```
   ssh root@<IP>
   curl -fsSL https://raw.githubusercontent.com/<OWNER>/radar-znyzhok/main/deploy/hetzner/setup.sh -o setup.sh
   less setup.sh          # прочитай, перш ніж виконувати чужий скрипт від root
   bash setup.sh
   ```
4. `nano /etc/radar/radar.env` — вписати `DATABASE_URL`, `RADAR_DOMAIN`, `GHCR_OWNER`,
   `BACKUP_TARGET`. **Ці значення не показуй CC і не клади в git.**
5. `git clone <репо> /opt/radar/repo`
6. `cd /opt/radar/repo/deploy/hetzner && docker compose --env-file /etc/radar/radar.env up -d`
7. **Перевір бекап одразу:**
   ```
   systemctl start radar-backup.service && journalctl -u radar-backup -n 30
   ```

## Що зроблено, щоб не повторити 17 липня

- `backup.sh` **падає**, якщо `BACKUP_TARGET` порожній. Мовчазний успіх гірший за помилку:
  інакше рік думаєш, що бекапи є.
- Перевіряє **розмір** дампа й **читає його назад** (`pg_restore --list`), шукаючи
  `price_snapshot`. Порожній дамп щодня — це катастрофа, яку помічають запізно.
- Бекап їде **поза цей сервер**: бекап поруч із базою — один пункт відмови.
- У compose немає `ports:` для API — назовні дивиться лише Caddy.
- Логи обмежені (`max-size`), інакше 40 ГБ диска тихо закінчаться.
- `unattended-upgrades` — сервер без автооновлень гниє незакритими дірками.

## Раз на місяць — обов'язково

**Віднови дамп у порожню базу й порахуй рядки.** Бекап, який жодного разу не
відновлювали, — це не бекап, а надія. Ми вже знаємо, скільки коштує різниця.
